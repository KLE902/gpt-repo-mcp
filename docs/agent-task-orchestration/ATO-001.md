# ATO-001 — Bounded Claude transport spike

Status: **ratified by owner on 2026-07-24 against reviewed commit `f9f536ab8b5e0781ec323da5b1b1ebd33ced3012`; this follow-up commit records that ratification mechanically**.

This is a one-off manual capability work package under the ratified [`DELIVERY_PLAN.md`](DELIVERY_PLAN.md). It does not establish a general work-package template, provider-neutral task schema, permanent Claude runner, or broader orchestration layer.

## 1. Capability question

Can one bounded read-only Claude task be started by ChatGPT through GPT Repo MCP and returned to ChatGPT through GPT Repo MCP with no owner prompt or result relay, while preserving exact repository, Git, context, runtime, and result boundaries strongly enough to justify Phase 1?

The spike measures transport value and host feasibility. It does not decide or implement the Phase 1 common durable lifecycle.

## 2. Trigger and owner-cost baseline

The owner reports at least **15 manual ChatGPT ↔ Claude transport events** during the recent project work and would probably have used the workflow more often if transport were faster and easier.

The dominant cost is not reliably reconstructable elapsed minutes. It is administrative energy and interruption:

- moving prompts and responses between tools;
- switching between computer and mobile;
- writing more on mobile than on a physical keyboard;
- attaching files from mobile, which is disproportionately cumbersome;
- losing momentum to the point that work may be abandoned for an extended period.

The retrospective active-time cost is therefore **not measured** and must not be invented. The spike records prospective owner effort, residual manual steps, and subjective administrative burden in addition to elapsed runtime.

## 5. In scope

- one fixed task file and one exact context declaration;
- one server-owned allowlisted Claude invocation or equally bounded temporary MCP path;
- current Claude CLI resolution, version, non-interactive authentication, capability, and structured-output verification;
- a fixed read-only tool boundary with no caller-supplied command, executable, arguments, provider flags, tools, environment, prompt, working directory, timeout, or cost setting at start time;
- bounded runtime, output, turns, and cost where the installed CLI supports reliable enforcement;
- deterministic local execution and result artifacts;
- MCP-owned parsing and validation of the actual current Claude output;
- explicit terminal classification for success, provider failure, output-contract failure, timeout, truncation, repository drift, context drift, or unverifiable read boundary;
- returning the validated result to ChatGPT through MCP;
- recording measured owner effort, elapsed time, residual steps, and administrative burden;
- at most one bounded repair attempt if the first end-to-end run exposes a narrow invocation or parsing defect.

- no refactor of the existing Codex create/start/review lifecycle;
- no permanent `ClaudeReadOnlyAdapter` or second durable Claude runner;
- no generic provider routing, dispatch command, broker, queue, dashboard, or task database;
- no generalized work-package template or requirement process;
- no Architecture Review automation;
- no specification drafting/critique workflow;
- no deterministic completion-claim framework;
- no Claude write access, Bash access, repository mutation, Git mutation, commit, push, pull request, or merge;
- no PKR product implementation or backlog reprioritization;
- no use of PKR-002 as the spike task;
- no automatic retry or recursive agent loop;
- no production hardening beyond what is required to run and classify this one spike safely.

Any attractive abstraction not required by the fixed end-to-end path is deferred.

## 7. Read and execution boundary

The implementation must fail closed unless all required evidence is present:

1. exact approved repository identity;
2. exact branch and 40-character HEAD;
3. clean index and worktree;
4. exact task-file identity;
5. exact declared context hashes;
6. server-owned Claude executable and argument construction;
7. allowlisted environment inheritance with secret-safe output;
8. no enabled write-capable Claude tools or unrestricted shell path;
9. bounded complete output and process-tree termination on timeout;
10. MCP-owned result parsing and validation;
11. unchanged branch, HEAD, index, worktree, and context hashes at result collection;
12. no concurrent MCP file, Git, task-execution, or other write operation against the pinned PKR worktree while the Claude read-only run is active.

A zero exit code, parseable JSON, or plausible narrative does not override a failed boundary check. A result invalidated by repository or context drift is retained only for diagnosis and must not be used as the PKR-004 position.

The live-worktree start-and-result race described in [`ARCHITECTURE.md`](ARCHITECTURE.md) must be either removed through an immutable snapshot or explicitly recorded as residual spike risk. It must not be silently described as full run-time immutability.

If the spike uses the live PKR worktree, MCP must acquire a bounded read lease or equivalent repository lock for the run. Known MCP write operations against that worktree must be refused until result collection completes. This controls known concurrent MCP writers without claiming protection against every external host process.

## 8. Required artifacts

The spike may choose exact local paths, but it must produce deterministic artifacts equivalent to:

- fixed human-readable task input;
- immutable run metadata containing repository, branch, HEAD, context identities, invocation profile, runtime bounds, and output contract;
- execution state with timestamps and terminal classification;
- bounded redacted provider output;
- validated structured result or explicit invalidation reason;
- measurement record for transport and owner effort.

The artifacts may be disposable after review. Their existence does not establish the Phase 1 contract.

## 9. Acceptance outcomes

### Transport success

All conditions must hold:

- ChatGPT starts the fixed Claude analysis without the owner copying or retyping the prompt;
- ChatGPT receives the complete validated result without the owner copying or retyping the response;
- owner prompt-relay count: `0`;
- owner result-relay count: `0`;
- no manual PowerShell, terminal, filesystem, or Claude UI action is required during the run;
- the result is retrievable through MCP after execution;
- the task and result are bound to the exact approved repository, branch, HEAD, and context hashes;
- the read-only and execution boundaries pass;
- repository and declared context remain unchanged;
- owner effort, elapsed time, residual steps, and administrative burden are recorded.

### Explicit boundary failure

The spike is still informative but not successful when execution is blocked with a precise fail-closed classification such as unsupported Claude capability, authentication failure, output-contract mismatch, unverifiable tool boundary, timeout, truncation, repository drift, or context drift.

### Value failure

The transport may work technically yet fail the product gate if owner administration is not materially reduced or the residual setup/coordination burden remains comparable to manual relay.

### Invalid result

The spike fails if the owner still transports the result manually, if the result is accepted despite missing boundary evidence, or if code expansion outruns the fixed task.

## 10. Measurements and decision record

Record:

- manual prompt relays by owner;
- manual result relays by owner;
- other manual owner actions;
- active owner administration time measured prospectively;
- end-to-end elapsed time;
- provider runtime;
- turns, usage, and reported cost where available;
- output completeness and validation result;
- repository/context boundary result;
- number and purpose of implementation attempts;
- residual setup steps that would recur in normal use;
- owner assessment: administrative burden **lower**, **similar**, or **higher** than manual relay;
- owner assessment: workflow should **proceed to Phase 1**, **proceed with bounded changes**, or **stop/simplify**.

The decision is based on measured transport value and boundary quality, not on code elegance.

## 11. Timebox and stop rules

- one initial bounded implementation and end-to-end attempt;
- at most one narrow repair attempt for a concrete host invocation or parser defect;
- no broad refactor during either attempt;
- stop immediately for repository/context drift, uncertain write capability, secret exposure, unbounded execution, or a requirement for caller-supplied commands;
- stop and return to owner decision if the fixed task cannot run without crossing a new security boundary;
- stop before Phase 1 design or implementation even if the spike succeeds.

1. Is PKR-004 an appropriately real but bounded dogfooding task, or does it introduce avoidable product coupling?

7. Is any material security, context, cost, or owner-decision boundary missing or incorrectly framed?
